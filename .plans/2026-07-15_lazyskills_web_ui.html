Context

LazySkills is a Go Bubble Tea TUI that scans, installs, updates, enables/disables, and removes agent skills across ~70 AI coding agents, backed by the skills.sh registry and the external skills CLI. The goal is to expose the same functionality through a browser UI, launched with a single npx command, that keeps the minimal terminal feel.

An import-level audit of the codebase shows it is already well-positioned for this:

Domain packages
scan, registry, actions, runner, locks, selfupdate, model, display — all UI-agnostic: no bubbletea/lipgloss imports, no stdout writes, JSON tags already on core types.
Trapped in TUI
Enable/disable shelf moves, lock pruning, broken-symlink deletion, bulk exec loop (internal/tui/update.go) and the git-clone discovery pipeline (internal/tui/discovery.go, ~530 lines).
Execution model
Mutations shell out to skills / npx skills and git; the server must run where the skills live.
State
skills-lock.json (project), ~/.agents/.skill-lock.json or XDG state (global), per-agent skill dirs, clone cache.
Dead end
The skills find action is Interactive (runs via tea.ExecProcess) — unusable from a web backend; replaced by the existing registry.Search HTTP client.

Recommendation

Recommendation

Ship a local-first web mode inside the existing Go binary: lazyskills web serves an embedded SPA plus a JSON/SSE API on 127.0.0.1. Distribute a thin npm wrapper package so npx lazyskills web downloads the right platform binary and launches it (the esbuild pattern). Tradeoff: we own binary distribution through npm in addition to brew/curl/go, in exchange for a zero-install browser entry point and one shared codebase for TUI and web.

A hosted web app is ruled out: everything LazySkills does is local filesystem reads/writes plus shelling out to local CLIs. Remote use still works via SSH port-forwarding the local server.

Architecture

One process, three layers. The browser talks to a loopback-only HTTP server; the server calls the same domain packages the TUI uses; mutations are executed by the existing runner against the local system.

The preview → confirm → execute flow the TUI already uses becomes wire protocol: the client asks for a CommandPreview (exact argv, Mutates, Dangerous flags), shows it to the user, then submits it for execution and streams output over SSE. Mutating jobs are serialized through a single-worker queue per project, replacing the TUI's implicit serialization.

Two properties the TUI got for free must become explicit protocol, because the web introduces multiple writers (two tabs, web + TUI, an agent installing skills mid-session):

  • Scan generations. Every scan response carries a generation number; previews embed it; execute rejects a preview built against a stale generation with 409, forcing the client to rescan and re-preview. A confirmed action can never run against a world that has changed underneath it.
  • Idempotency keys. Execute requests carry a client-generated key; resubmission (double click, tab reconnect, SSE retry) returns the existing job instead of running the command twice.

Scope guard

cwd comes from where the server was launched, exactly like the TUI. The client never sends filesystem paths in v1 — multi-project switching is a deliberate later feature, not an accidental path-traversal surface.

UI guidelines

The web UI should read as "the TUI, rendered by a browser" — not a SaaS dashboard. Users who know the terminal app should feel zero relearning cost; the browser earns its place only through better density, real tables, and mouse support.

Layout

  • Same three-region shell as the TUI: a slim top bar (project path, scope, agent filter, search), a two-pane body (left: skill list grouped by source; right: detail pane), and a one-line status footer showing counts, active filters, and key hints — the web equivalent of the TUI footer.
  • Fixed full-viewport app layout, panes scroll internally. No page scroll, no hero, no marketing chrome, no sidebar navigation. The app is the two panes.
  • Detail pane tabs mirror the TUI detail view: metadata, rendered SKILL.md, health issues, and the per-agent visibility matrix. The visibility matrix (skills × agents) is the one place the web should beat the TUI — render it as a real table.
  • Narrow viewports collapse to a single pane with the detail view pushed as an overlay.

Visual language

  • Dark only. Near-black background, panels one step lighter, 1px borders. No shadows, no gradients, no glassmorphism, no card grids, no icons beyond a few status glyphs (● ✓ ✗ ⚠).
  • Monospace where the TUI is monospace: skill names, paths, commands, lock data, agent IDs. A UI sans face only for descriptions and prose.
  • One accent color (the brand cyan, #3dbbff) reserved for selection and focus, plus semantic green/amber/red strictly for health and visibility states. Everything else is grayscale.
  • Compact density: 13–14px base, list rows ~28px tall, tabular numerals. Closer to lazygit than to Linear.
  • Empty and loading states are plain text lines ("scanning…", "no skills match"), not spinners or illustrations.

Interaction

  • Keyboard-first, same keymap as the TUI: / search, a cycle agent filter, f scope toggle, enter details, c actions, space select, u update, x remove, ? keymap overlay. Mouse is additive, never required, and every mouse affordance shows its key hint.
  • Actions always go preview → confirm: the action modal shows the exact command that will run, verbatim, in a code block — identical to the TUI's command preview. Destructive actions get a distinct red confirm.
  • Command output streams live into a bottom drawer styled like a terminal (monospace, no wrapping tricks), then the list auto-refreshes from a new scan.
  • Bulk selection uses space + checkboxes; the footer shows the pending selection count and available bulk actions.
  • Rendered SKILL.md is sanitized before injection — skill content is untrusted third-party input.

Phases & end criteria

Phase 0 — Domain extraction prerequisite

Hoist the business logic currently trapped in the TUI into UI-agnostic packages: enable/disable shelf moves, lock pruning, broken-symlink deletion, post-remove lock cleanup, and the bulk-exec loop into a new internal/skillops; the git clone/cache/discovery pipeline from internal/tui/discovery.go into internal/discovery. Rewire the TUI to call them. Pure behavior-preserving refactor.

Done when
  • internal/skillops and internal/discovery exist with unit tests; neither imports bubbletea/lipgloss.
  • internal/tui contains no direct os.Rename/os.Remove mutation logic and no git invocation code.
  • go build ./... && go test ./... green; go vet clean.
  • Manual TUI smoke pass unchanged: scan, disable/enable, prune lock, delete broken symlink, single + bulk remove, source discovery.

Phase 1 — HTTP API + lazyskills web subcommand

New internal/webserver package and web subcommand: lazyskills web [--port 7777] [--open] [--read-only]. Implements the API table above, the single-worker mutation job queue with scan generations and idempotency keys, SSE job streaming, and the full security model (loopback bind, startup token, Origin/Host checks).

This phase also owns the child-process policy. In the TUI, when skills/npx prompts ("Ok to proceed? (y)", first-run questions), the user answers; over SSE there is no stdin and the job would hang forever. Jobs therefore run with stdin closed, every available non-interactive/--yes flag set, a hard timeout with process-group kill, and a stall detector that surfaces "this command appears to be waiting for input" instead of silence. First-run npx skills package downloads are long and noisy — stream them honestly with a "downloading dependency" state rather than looking frozen.

Done when
  • Every endpoint in the API table responds correctly via curl with the token; handler-level integration tests cover scan, preview, execute, and job streaming.
  • Requests without the token get 401; requests with a non-localhost Origin or Host header get 403 (unless explicitly allow-listed, see security model); ss -tlnp confirms the listener is bound to loopback only, and the printed URL connects whether the browser resolves localhost to 127.0.0.1 or ::1.
  • Two concurrent execute requests are provably serialized (second waits in queue), and job output arrives over SSE followed by a completion event.
  • A test fixture command that prompts on stdin does not hang: the job fails within the timeout with a "waiting for input" diagnosis in the SSE stream. Killing a job kills its whole process group.
  • Executing a preview built against a stale scan generation returns 409; the integration test mutates a skill dir between preview and execute to prove it.
  • Resubmitting an execute request with the same idempotency key returns the original job ID; the command runs exactly once.
  • --read-only rejects all mutating endpoints with 403.

Phase 2 — Embedded SPA with TUI parity

Svelte + TypeScript + Tailwind SPA implementing the UI guidelines above, built by Vite, embedded with go:embed, served by the Phase 1 server. Feature parity with the TUI is the bar.

Done when — parity checklist all green
  • List grouped by source; search (/); agent filter cycle (a); project/global/all scope toggle (f); detail view with metadata, rendered + sanitized SKILL.md, health issues, and the agent visibility matrix.
  • All actions work end-to-end with preview → confirm → streamed output → auto-rescan: install (registry search + source discovery), reinstall/update, remove, enable/disable, prune locks, delete broken symlink, single and bulk.
  • Update-available banner from /api/update; ? keymap overlay; all TUI keybindings functional.
  • A single go build binary serves the complete UI with zero external files or network assets (verified offline).
  • Sanitization test passes against a hostile SKILL.md fixture (script tags, event handlers, javascript: URLs stripped).

Phase 3 — npx distribution

Publish an npm wrapper so npx lazyskills web works with nothing preinstalled: a tiny launcher package plus per-platform optionalDependencies packages carrying the Go binary (the esbuild pattern), wired into the existing release CI.

Done when
  • npx lazyskills@latest web launches the UI and opens the browser on macOS (arm64 + x64), Linux x64, and Windows x64, on machines without Go or brew.
  • The launcher resolves the platform package, falls back to a checksum-verified download from GitHub releases if optionalDependencies were skipped, and fails with a clear message on unsupported platforms.
  • npm publish is automated in release CI; npm package version and GitHub release tag are always identical (CI asserts this).
  • README documents the npx path alongside brew/curl/go.

Phase 4 — Polish & parity extras

Quality-of-life work once the core ships: fsnotify-driven live refresh pushed over SSE, copy-path buttons replacing "open in editor", demo media, and any parity gaps found in real use.

Done when
  • External changes to skill dirs appear in the UI within 2s without manual refresh (if fsnotify is in scope per the tuner below).
  • README web section with screenshot/gif; --read-only documented.
  • No open parity issues labeled web-parity in the tracker.

npx distribution detail

The npm side is deliberately dumb — all product code stays in Go. Package layout:

# launcher (what users type)
lazyskills                      # bin: launcher.js — resolve platform pkg, spawn binary, pass args through

# per-platform binary packages (optionalDependencies of the launcher)
@lazyskills/darwin-arm64
@lazyskills/darwin-x64
@lazyskills/linux-x64
@lazyskills/linux-arm64
@lazyskills/win32-x64

The launcher checks process.platform/process.arch, finds the installed platform package, and spawns the binary with the user's argv — so npx lazyskills web, npx lazyskills find --json "browser automation", and even the plain TUI all work through npm. If the platform package is missing (some package managers skip optionalDependencies), it downloads the matching release artifact from GitHub and verifies the published SHA-256 before executing.

Reserve the name now

Name-squatting is a race that starts the moment this plan is public. Publish a stub lazyskills package (and the @lazyskills org scope) immediately — before any announcement — not "before Phase 3". If the unscoped name is already taken, fall back to the scope for the launcher too (npx @lazyskills/cli web).

Security model

This API executes shell commands; an exposed endpoint is remote code execution. These are non-negotiable and land in Phase 1, not as a hardening pass later:

  • Loopback bind only. Listen on both loopbacks (127.0.0.1 and ::1) — browsers that resolve localhost to ::1 otherwise get connection refused — and print the literal-IP URL. No flag to bind wider in v1; remote use is SSH port-forwarding.
  • Startup token, then cookie. Random per-launch token embedded in the URL the CLI prints and opens; on first load the SPA exchanges it for an HttpOnly SameSite=Strict session cookie and strips the token from the URL, so it doesn't live in browser history for the lifetime of a long-running server. All requests including SSE require the cookie (or the token, for curl use).
  • DNS-rebinding defense, with a deliberate remote-dev stance. Reject any request whose Host isn't localhost and any state-changing request whose Origin isn't the server's own origin. This intentionally breaks Codespaces / VS Code port forwarding / devcontainer proxies, which rewrite both headers — for those users, provide an explicit, loud escape hatch (--allow-origin https://xyz-7777.app.github.dev) rather than loosening the default. Document this in the README; it will otherwise surface as mystery 403 bug reports from remote-dev users.
  • No client-supplied paths or argv. The client sends action identifiers and skill IDs; the server rebuilds commands through actions.* with its existing sanitization. Execute accepts a preview hash, not a command string.
  • Untrusted content. SKILL.md and registry metadata are sanitized before rendering (XSS), reusing the existing compat sanitizers server-side plus DOM sanitization client-side.
  • --read-only for cautious or demo use: scan and search work, every mutation returns 403.

Risks

Risk Severity Mitigation
API exposes command execution beyond localhost high Loopback bind + token + Origin/Host checks land in Phase 1 with explicit end criteria; no wide-bind flag exists.
Phase 0 refactor silently changes TUI behavior medium Behavior-preserving diff discipline, existing test suite, scripted manual smoke pass in the end criteria.
Malicious SKILL.md achieves XSS in the browser medium Server-side sanitization + client DOM sanitizer, hostile-fixture test required by Phase 2 end criteria.
npm name unavailable / npx cold-start feels slow medium Reserve the npm name/scope with a stub package now, before announcing; platform packages make cold-start a single small binary fetch; local npx cache makes reruns instant.
Server env ≠ user shell env (agents/paths differ) low Launcher spawns the binary from the user's shell, inheriting env exactly like the TUI does today.

Verification

Beyond per-phase end criteria, the release gate for the first web version:

  • Cold-machine test: on a VM with only Node installed, npx lazyskills web opens the browser and a full install→update→disable→remove cycle succeeds against a scratch project.
  • Security probe script in CI: tokenless request, spoofed Origin, spoofed Host, non-loopback connection attempt, and a path-traversal payload against every endpoint — all rejected.
  • Parity audit: walk the README key table (/ a f enter c n space u x ?) in the web UI; every row must have a working equivalent.
  • TUI regression: full existing test suite plus manual smoke on the TUI, confirming the web work changed nothing for terminal users.

Open questions

  • Is the lazyskills npm package name available, and do we want the unscoped name or the @lazyskills scope? (Blocks Phase 3 naming only.)
  • Svelte is proposed for bundle size and simplicity — any preference for React for contributor familiarity?
  • Should lazyskills web also be advertised from inside the TUI (a footer hint), or stay a README-level feature at launch?
  • Default port: fixed (memorable, e.g. 7777) or ephemeral (collision-free)? Proposal: cwd-hash-derived default with automatic fallback to ephemeral.
  • Parity governance: is the web UI a first-class surface (new features block until both TUI and web ship) or best-effort (web may lag behind a web-parity label)? This should be decided and written down before Phase 2 starts.
